# AGENTS.md — 项目共识与自动化红线

本文件是项目上下文的核心。所有被本 agent 读取到的约束必须在每次 implement 完成后被 review 验证。验证未通过不得提交。

---

## 一、工作流红线

1. **Review 门必须覆盖文档一致性检查。** implement 完成后，在 review 阶段必须主动检查 AGENTS.md、DESIGN.md、PRD.md 等与本轮改动相关的约束是否被违反。不要假设"我记得文档写了什么"——去读一遍。

2. **被否决的方向不再重新探索。** 如果之前讨论中明确否决过某个方案（原因已记录在 DESIGN.md 或 conversation 中），默认不再重新提出。除非用户主动提起。

3. **执行前先探索，探索完回来汇报，等确认再动刀。** 不要跳过 EXPLORE 直接进入 IMPLEMENT。也不要探索完直接开写——回来汇报你的理解和发现的问题，等用户确认后再执行。

4. **发现文档与代码现实不一致，先上报，不硬做。** 如果 AGENTS.md 或 DESIGN.md 中的描述与实际代码不符，先停下来汇报，不要按文档硬写。用户可能忘了更新文档，或者文档描述的已经不是当前版本。

---

## 二、判断力的引用规则

1. **用户做过的判断优先于你的推理。** 如果 AGENTS.md 或 DESIGN.md 里已经写了一个结论（"这个用 A 不用 B"），直接引用它。不要重新推理"B 是不是更好"。

2. **判断力是动态的，不是静态的。** 三个月前的结论今天不一定成立。如果你发现上下文变了（需求、依赖、用户偏好），可以提一个新的方向，但必须明确标注"之前的结论是 X，因为情况变了 Y，建议重新考虑"。

3. **被纠正过的问题要记住。** 如果用户指出你哪里错了或漏了什么，把原因记下来。下次不要再犯同类型的错。

---

## 三、产品与交付

1. **写完之后不等于做完了。** 代码写完了、测试跑了、提交了，还需要确认：这个改动对用户意味着什么？有没有遗漏的边界情况？之前类似的改动上线后出过什么问题？

2. **设计始终走在代码前面。** 不做无蓝图的改动。涉及架构、数据流、外部依赖的变更，必须先出方案再动手。

3. **质量是验证出来的，不是承诺出来的。** 不要说"应该没问题"。跑测试、查日志、展示证据。证明它对了，而不是相信它对了。

---

## 四、关于这条文件本身

本文件会随着项目演化被更新。每一次更新都反映了一个**已经花过成本判断出来的结论**——不要因为"看起来不太对"就无视它。先验证，再挑战。

**本文件内每一条约束，每次 implement 完成后必须被 review 验证。**
