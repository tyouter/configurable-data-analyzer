Understanding pandas — the Excel reader

When the analyzer needs to read your data file, it uses a Python library called pandas.

CODE 

def load_data(self) -> pd.DataFrame:
    """Load data from Excel file"""
    if not os.path.exists(self.data_path):
        raise FileNotFoundError(f"Data file not found")

    self.data = pd.read_excel(data_path)

    # Parse timestamps
    if 'strt_time_nano' in self.data.columns:
        self.data['strt_time_dt'] = pd.to_datetime(
            self.data['strt_time_nano'], errors='coerce')
PLAIN ENGLISH

This function loads data from an Excel file...

First, check if the file actually exists — don't crash if it's missing

Use pandas read_excel() to load the whole spreadsheet into memory...

Then convert the timestamp column to actual datetime objects so we can work with dates properly

💡
Key Insight

Libraries like pandas are what make Python powerful for data. Instead of writing code to read files line by line, you just call one function and get a complete data structure (DataFrame) that you can filter, sort, and calculate on. Engineers call this "don't reinvent the wheel."

KPI Metrics — How the Numbers Are Calculated

What is a KPI?

KPI (Key Performance Indicator) is like a report card grade for your product. It tells you, at a glance, how things are going.

👥

app_active_rate

25.23% — What percentage of users actually use the app (stayed longer than 5 seconds)

⏱

app_retention_rate

24.3% — How many users came back 7 days later (shows if product keeps people engaged)

🎯

ai_guide_usage_rate

12.7% — How often people click the AI travel guide button (measures feature adoption)

💡
Key Insight

KPIs are "leading indicators" in business. They condense complex data into single numbers you can track over time. Good KPIs are actionable — if retention drops, you know to investigate why users aren't coming back.

How different KPI types are calculated

The analyzer supports several types of calculations. Each type describes a different "recipe" for crunching the numbers:

🔢
count_unique_users

Simply count how many different values appear in a column (like counting names in a list without duplicates)

📊
count_events

Count every row that matches a pattern — total clicks, total opens, total views

⏱
ratio

Divide one KPI by another — like "app_avg_opens_per_user" is "total_opens" divided by "active_users"

⏰
exposure_time

Add up time differences between when people opened something and when they closed it — measures how long they spent

📉
retention_rate

Check if users who appeared on day X also appeared on day X + 7 — do they stick around?

See a KPI calculation in code

Here's how the analyzer calculates "app_active_rate" — the percentage of active users. Watch the code on the left and plain English on the right:

CODE 

elif kpi_type == 'active_rate':
    event = kpi_config.get('event')
    min_duration = kpi_config.get('min_duration_seconds', 0)
    active_users = self.calculate_active_users(event, min_duration_seconds=min_duration, **kwargs)
    total_users = self.data['reduser_id'].dropna().nunique()
    return round((active_users / total_users * 100) if total_users > 0 else 0, 2)
PLAIN ENGLISH

If the KPI type is "active_rate" (we're measuring active users)...

Get which event to track (like "discovery_page_pageshow")...

Get minimum time to count as "active" (at least 5 seconds)...

Calculate how many users spent more than 5 seconds...

Divide by total unique users in the dataset...

Multiply by 100 to get a percentage, round to 2 decimal places

📘
Good to know

The nunique() method removes duplicate users (if someone opened the app twice, they're counted once). dropna() ignores missing values to avoid errors.

YAML — the recipe book for KPIs

All KPI definitions are stored in YAML files. This is how the analyzer knows HOW to calculate each KPI:

YAML CONFIG

Define a KPI named "app_active_rate"

Give it a friendly name like "APP 活跃率"

Set type to "active_rate" — this tells the analyzer which calculation recipe to use

Specify the event to track: "discovery_page_pageshow"

Set minimum duration to 5 seconds (only count users who stayed at least this long)

Add a description in Chinese explaining what this measures

CODE 
app_active_rate:
  name: "APP 活跃率"
  type: "active_rate"
  event: "discovery_page_pageshow"
  min_duration_seconds: 5
  description: "至少一次有效 APP 操作的活跃用户数量..."
💡
Key Insight

Separating configuration (WHAT to calculate) from code (HOW to calculate it) is a crucial pattern. It means you can add new KPIs just by editing the YAML file — no code changes needed. Engineers call this "configuration over hardcoding."

Extending the Analyzer — Adding Your Own Features

The tool is built to be extended

One of the most important design decisions is that you can add new KPIs and analysis methods without touching the core code. This is done through a few extension points:

1
Edit metrics.yaml

Add a new KPI definition in your project's config file and it just works — new metrics calculated automatically

2
Add new analysis methods

Write a new method in the RednoteAnalyzer class (like analyze_video_behavior()) and call it from your code

3
Create new projects

Use ConfigManager to create a new project with its own schema and metrics for a completely different dataset

Adding a new KPI — a practical example

Let's say you want to track a new metric: "how many users shared content this week?" Here's how you'd add it:

YAML ADDITION 
share_metrics:
  # Add this to existing section
  share_usage_count:
    name: "内容分享使用次数"
    type: "count_events"
    events: ["post_detail_page_share_button_click"]
    description: "统计周期内分享按钮点击次数"
WHAT YOU DID

Added to the "share_metrics" category (or created a new one)

Named the metric "share_usage_count" in English (for code)

Set type to "count_events" — reuse existing calculation method!

Listed which event(s) to count — the share button click

Added a Chinese description for human readability

That's it! Next time the analyzer runs, this KPI is automatically calculated. No code changes needed.

Adding a custom analysis method

Sometimes you need something more complex than counting events. You can add new methods to the analyzer class:

NEW METHOD 
def analyze_top_content(self) -> Dict[str, Any]:
    """Analyze most popular content"""
    if self.data is None:
        self.load_data()

    content_events = self.data[
        self.data['rednote_post_is_like'].notna()
    ]

    top_posts = content_events[
        self.data['rednote_post_id']
    ].value_counts().head(10)

    return {
        'top_posts': top_posts.to_dict(),
        'total_likes': len(content_events)
    }
WHAT IT DOES

Define a new function that returns a dictionary of results

Filter for rows where users liked content (is_like column is not empty)

Grab the post IDs from those rows

Count how many times each post appears (value_counts) and get top 10

Return both the ranking and total like count

📘
Good to know

Pandas DataFrame methods like value_counts() make aggregation easy. They automatically count frequencies and sort, saving you from writing loops to do the same thing.

Understanding ProjectMatcher — how it knows your data

How does the analyzer automatically figure out which configuration to use? It uses smart matching:

📂
Filename pattern matching

If your file is named "rednote_data_202403.xlsx", it looks for a pattern like "rednote_.*" in the config

📋
Column signature matching

Checks if your data has specific required columns like "user_id" or "event_type" — like checking if a document has the right headers

🎯
Similarity scoring

If the first two methods fail, it calculates Jaccard similarity (how many column names overlap) and picks the closest match

💡
Key Insight

This multi-strategy approach makes the tool flexible. You can rename files, add columns, or reorganize folders — and the matcher will still find the right configuration most of the time.
