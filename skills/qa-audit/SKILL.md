# Dashboard Quality Gate — Commercial-Grade Audit Protocol

本协议不是 checklist，而是**侦查方法论**。覆盖从数据接入到最终展示的全链路，适配电商、金融、医疗、游戏、SaaS、社交、出行等任何行业。

Agent 应以**专业数据分析师**的身份，对每个 Dashboard 执行三层审计：

```
Layer 1: Data Integrity（数据完整性）
Layer 2: Computational Rigor（计算严谨性）
Layer 3: Visual Professionalism（视觉专业性）
```

每一层的问题分三类处理：
- **⚠️ AUTO-FIX**：Agent 可直接修正（如 SQL 语法、列名可读性）
- **🔔 REPORT**：报告给用户，由用户决定怎么修（如异常用户处理、口径选择）
- **🚨 FLAG**：标记但不阻断，仅作为观察记录写入审计报告

# Layer 1: Data Integrity（数据完整性）

## 1.1 数据范围检查

**目的**：确保引用的数据覆盖了预期的时空范围。

检查方法：
- 对每个数据源的日期字段，检查 `MIN(event_date)` 到 `MAX(event_date)` 的覆盖度
- 日期是否有预期之外的断档（如周一到周四有数据，周五到周日全空 → 可能是采集断点）
- 最新数据与当前时间的时差（如数据停留在 3 天前 → 延迟问题）
- 维度覆盖：如按地域分析却只有一个省的数据，按平台分析却只有一种平台

**行业无关模式**：所有包含时间字段的分析，第一步永远是时间范围扫描。

## 1.2 缺失数据处理

**目的**：识别可能扭曲结果的空值分布。

检查方法：
- 对每个参与聚合计算的字段，检查 `null_rate`
- 高缺失率（>30%）字段参与计算时，标记结果可靠性
- 区分「自然稀疏」（如用户只有在点击时才有的字段，多数行为空）和「数据丢失」
  - 自然稀疏：该字段仅在特定事件中出现（如订单金额只有支付事件才有）
  - 数据丢失：该字段在所有事件中都有定义但大量为空
- 跨表/跨事件 JOIN 时的 NULL 处理方式（INNER JOIN vs LEFT JOIN 的选择）

**报告**：列出高缺失字段、缺失率、参与计算的指标、对结果的影响估计。

## 1.3 异常值检测

**目的**：识别数据中的极端值或测试数据。

检查方法（通用统计方法，不限特定字段或阈值）：

- **IQR 方法**：对数值型指标，计算 Q1 / Q3 / IQR，标记 `> Q3 + 1.5*IQR` 或 `< Q1 - 1.5*IQR` 的值
- **Z-Score 方法**：对近似正态分布的数据，标记 `|z| > 3` 的值
- **MAD 方法**（对偏态分布更鲁棒）：标记 `> median + 3 * MAD` 的值
- **量级断崖检测**：对聚合结果（如 `COUNT(*) GROUP BY user_id`），检查 top N 的值是否出现断崖式下降（如序列为 5453, 3325, 2265, 2003, ... 中间有明显差距）

**AUTO-FIX 条件**：当异常值可被明确归因为测试/监控流量时（如 user_id 包含 "test"、"dev" 等关键词，或事件行为模式明显异常），Agent 可自行添加过滤条件。

**REPORT**：无法明确归因的异常，向用户报告数据分布，由用户判断处理方式。

## 1.4 数据分布形态检查

**目的**：识别数据分布的特性，指导后续的统计处理和可视化选择。

检查方法：
- **偏度检查**：`mean - median` 的差异。大幅右偏 → 中位数比均值更代表典型值
- **峰度感知**：极端值的频率是否异常高
- **稀疏性检查**：group by 后的分组数量 vs 数据量的比例（如 65K 事件分布在 297 个用户上，平均每个用户 220 事件）
- **零膨胀检查**：指标值为 0 或 NULL 的比例过高（>50%）

**用途**：对偏态分布的数据，用中位数而非均值做箱线图中心线；对零膨胀数据，考虑使用两层模型（是否发生 + 发生多少）。

## 1.5 重复数据检测

**目的**：防止重复记录导致的聚合膨胀。

检查方法：
- 对关键 ID 组合（如 `trace_id + span_id` 或事件唯一标识）检查重复率
- 如果存在完全重复的行，计算对指标的影响
- 对 `COUNT(DISTINCT)` 聚合的保护：检查是否需要改为 `COUNT(DISTINCT)` 而非 `COUNT(*)`

# Layer 2: Computational Rigor（计算严谨性）

## 2.1 比率指标验证

**目的**：确保所有比率（率、占比、渗透率、增长率、留存率等）的计算逻辑正确。

检查方法：
- 对每个结果在 0~100 或 0~1 范围内的数值型指标，检查 SQL 中是否存在对应的除法运算
- 如果数值在 0~100 范围内但 SQL 中没有除法 → 🔔 标记为可能的绝对计数而非比率
- 除法的分母是否可能为 0 → `NULLIF(denominator, 0)` 保护
- 分子 > 分母 的情况 → 标记为不可能比率（除非大于 100% 是合理的，如增长率）
- 留存率/复购率的 COHORT 计算是否正确：第 N 天的留存用户 ≠ 第 0 天的活跃用户

**AUTO-FIX**：遗漏的 `NULLIF` 分母保护可以自动添加。

## 2.2 聚合粒度一致性

**目的**：确保不同指标在聚合时使用了相同的粒度约束。

检查方法：
- 参与同一张图或同一 dashboard 的指标，其 WHERE 条件（排除规则、时间范围）是否一致
- `COUNT(DISTINCT user_id)` vs `COUNT(DISTINCT device_id)` — 不同粒度的对比需确认口径
- 每天每用户"最多一次"的限制（如打开次数）是否实现为 `COUNT(DISTINCT date || ':' || user_id)`
- 累加性检查：`7 天总和 ≠ 日数据的 SUM()` 时，是因为去重逻辑不同还是数据问题

## 2.3 时间序列计算正确性

**目的**：环比、同比、累计等时序计算逻辑验证。

检查方法：
- **环比（WoW / DoD）**：第一天是否处理为 0 而非 `(第一天值-0)/0` 导致的除零或无穷大
- **同比（YoY）**：是否与去年同期同周/同月对比（而非简单的前 365 天）
- **累计（MTD / YTD）**：累加范围是否正确
- **移动平均**：窗口边界处理是否正确（前 N 天包含当天吗）
- **时序差计算**（如使用时长）：`end_time - start_time` 是同一事件的开始到结束，还是上一个事件结束到下一个事件开始？两种语义不同

**AUTO-FIX**：增长率首日设 0。

## 2.4 加权平均 vs 简单平均

**目的**：防止在需要加权平均时错误使用简单平均。

检查方法：
- 当天均指标（如人均时长、客单价）是否使用 `SUM(value) / COUNT(DISTINCT user)` → 正确
- 周/月均指标是否直接平局了日均值（正确方式：周总值 / 周用户数，而非 `AVG(日指标)`）
- 对聚合的聚合（avg of avg）保持警惕

## 2.5 统计显著性提示

**目的**：对样本量不足的指标给出警示。

检查方法：
- 分组后的每组样本量 < 30 → 🔔 标记"小样本，统计意义上建议谨慎解读"
- 留存率计算中，base cohort 人数 < 50 → 🔔 标记"cohort 基数较小，留存率波动可能较大"
- 转化漏斗中，顶部环节人数 < 100 → 🔔 标记"顶部流量少，后续转化数据仅供参考"
- 某个指标只有 1~3 个数据点 → 🚨 标记为"趋势无法判断"

# Layer 3: Visual Professionalism（视觉专业性）

## 3.1 渲染完整性

**目的**：确保所有图表被正确渲染，无回退。

检查方法：
- `canvas_count == scpec_chart_count - kpi_card_charts`
- `table_elements == 0`（table 意味着 ECharts 渲染失败，回退为数据表）
- `js_errors == 0`
- 如果上述不满足，逐个列出失败的 chart_id 和失败原因

**AUTO-FIX**：根据失败原因修正 spec 中的 SQL 列格式。

## 3.2 Y 轴刻度完整性

**目的**：确保 Y 轴刻度没有误导性截断。

检查方法：
- bar / line / bar_line 等轴类图，检查 `yAxis[0].min` 和 `yAxis[0].max`
- 如果 `min > 0`，检查是否为了放大差异而刻意截断起点 → 🚨 标记为"Y 轴非零起点，可能夸大差异"
- 如果 `min` 和 `max` 未设置（自动缩放），计算数据范围与轴范围的比值
- 对箱线图：`max / q3 > 5` 时，检查是否有 outlier 导致 Y 轴被拉伸 → 🔔 报告

**AUTO-FIX**：箱线图 outlier 问题 → 在 SQL 中增加阈值上限。

## 3.3 双轴图左右轴协调性

**目的**：确保双轴图的左右轴量级匹配，不产生视觉误导。

检查方法：
- 对 `bar_line` 类型，检查左右两个 Y 轴的 `max` 比值
- 如果左轴 max 是 50，右轴 max 是 300%，两者的视觉缩放比例是否合理
- 左右轴的 0 点是否对齐（如果左轴从 0 开始而右轴从 20% 开始，柱子和线的起点不同会产生错觉）
- 线系列的值域如果远远窄于柱系列的值域，线的趋势会被压平 → 考虑是否应该使用双轴，还是分别展示

## 3.4 标签与文本可读性

**目的**：确保所有文本元素清晰可读。

检查方法：
- **X 轴标签旋转**：分类标签超过 8 个时是否自动旋转 45°，防止重叠
- **图例位置**：legend 是否与图表内容重叠（检查 legend 的 top/right 偏移是否足够）
- **KPI 卡片**：数值是否过大导致溢出（千位分隔符）
- **百分比单位**：比率类数值是否在视觉上体现为百分比（Y 轴 label format、KPI 卡片后缀）
- **颜色区分**：多 series 的颜色在灰度下是否可区分（色盲友好性）

## 3.5 图表类型适用性

**目的**：评估 chart_type 是否与数据特性匹配。

检查方式：
- **时序数据** → line 或 area（bar 也接受但有更好的选择）→ 不应该用 pie
- **组成数据**（各部分占比）→ pie 或 stacked bar → 不应该用多独立 bar
- **排名数据** → ranking_bar（水平）→ 不应该用垂直 bar
- **分布数据** → boxplot 或 histogram → 不应该用 pie
- **对比数据**（类别间比较）→ bar
- **大于 7 个扇区的 pie** → 改为 bar（信息密度过高）

**REPORT**：向用户建议更合适的图表类型（不自动改，除非用户同意）。

## 3.6 数据标签与精度

**目的**：确保数值标签的显示精度合理。

检查方法：
- 指标的 `kpi_format` 与显示精度是否匹配（percent 显示一位小数，integer 不显示小数）
- 饼图的扇区标签是否默认显示百分比（formatter 应为 `'{d}%'` 或 `'{b}: {d}%'`）
- 排名图的数值标签是否默认显示（position: right）
- 过多的小数位（如增长率显示 28.5714285714%）是否截断到 1~2 位小数

   - [实体ID]：N 事件 / 均值 M | 建议：XXX

## 执行流程

```
指标注入完成
    ↓
第一阶段：Layer 1 审计 → REPORT ⚠️ / 🔔 → 用户对齐 → 修正
    ↓
写 spec → generate_dashboard_from_spec
    ↓
第二阶段：Layer 2 + 3 审计
    ├── AUTO-FIX 项直接修
    ├── REPORT 项向用户展示
    └── FLAG 项记入审计报告
    ↓
重新生成 → 再审计 → 直到 Layer 2+3 全部通过
    ↓
第三阶段：业务层观察（仅 FLAG，不主动修）
    ↓
✅ Dashboard 交付 + 审计报告
```
